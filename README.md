# AI 챗봇 서비스 MVP (Sionic AI 과제)

## 🎯 가장 구현하기 어려웠던 기능
**"Spring WebFlux 기반 스트리밍 응답 구현 시, 트랜잭션과 스레드의 생명주기 제어"**
- **어려웠던 점**: OpenAI API의 응답을 클라이언트에게 실시간 스트리밍(SSE)으로 중계할 때, 메인 스레드 고갈을 막으면서도 DB 트랜잭션을 안전하게 관리하는 아키텍처를 설계하는 것이 가장 까다로웠습니다.
- **해결 과정에서의 고민**: 
  - 스트리밍 도중 클라이언트가 강제 이탈하거나 네트워크가 끊어지는 수많은 엣지 케이스(Edge Case)를 어디까지 방어해야 할지 결정하는 것이 어려웠습니다. 결과적으로 WebFlux의 `.doFinally`를 활용해, 비정상 종료 시에도 그때까지 수신된 조각 데이터(Partial Response)를 백그라운드 스레드에서 안전하게 DB에 저장하도록 Fail-safe 로직을 구축했습니다.

## 🤖 AI 활용 방식
- **활용 방식**: 
  - **코드 구현**: 구체적인 구현 방향과 핵심 아키텍처 설계는 제가 직접 주도하고, 단순 코드 타이핑은 AI에게 위임했습니다. 이후 AI가 생성한 코드의 기능별 플로우를 꼼꼼히 점검하며, 아키텍처 의도와 다르게 구현된 점을 직접 지적하고 수정하는 방식으로 활용했습니다.
  - **문서화 및 회고**: 구현 과정에서 엣지 케이스나 한계점 등 핵심 포인트를 식별할 때마다, AI를 활용해 `README.md`의 '향후 개선 방향' 등에 문서를 작성하는 데에도 적극 활용했습니다.
- **실제 점검 및 수정 사례 (Spring WebFlux 블로킹 이슈)**: 
  - AI가 초기에 작성한 채팅 생성 로직에서는 `Flux` 스트림을 리턴하기 전에 JPA 엔티티를 조회/저장하는 동기(Blocking) 코드가 메인 스레드에 그대로 노출되어 있었습니다.
  - 이를 제가 발견하고 "이대로면 Netty 이벤트 루프가 DB I/O에 의해 블로킹되어 리액티브의 의미가 퇴색된다"고 지적했습니다. 이후 해당 블로킹 구간을 `Mono.fromCallable { ... }.subscribeOn(Schedulers.boundedElastic())`으로 격리하여 실행하도록 로직을 수정했습니다.

## 📌 주요 기술 스펙 및 아키텍처 요약
1. **논블로킹(Non-blocking) 비동기 처리**: `WebClient`와 `Flux`를 활용하여 OpenAI 스트리밍 연동 시 메인 스레드 블로킹을 방지했습니다.
2. **트랜잭션 분리 및 연결 관리**: 외부 API 통신과 내부 JPA 트랜잭션 로직의 스코프를 분리했습니다. 또한, 클라이언트의 비정상 연결 종료(CANCEL) 시 `.doFinally` 블록을 통해 생성된 부분 응답(Partial Response)이 유실되지 않도록 저장합니다.
3. **엔티티 모델링**: JPA 단일 테이블 상속(`SINGLE_TABLE`) 전략을 통해 사용자 채팅과 AI 채팅을 분리했으며, `parent_chat_id` 자기 참조 매핑을 적용하여 질문과 답변의 인과관계를 구조적으로 기록했습니다.
4. **데이터 정합성 및 쿼리 최적화**: 스레드 목록 조회 시 `IN` 쿼리를 활용해 N+1 문제를 방지했으며, 피드백 생성 등 동시성 이슈가 예상되는 지점에는 복합 유니크 조건을 걸어 DB 레벨에서 레이스 컨디션을 방어했습니다.

## ⚠️ 한계점 및 향후 개선 방향
1. **스레드 생성 동시성 처리**: 유저가 동시 다발적으로 첫 채팅 요청 시 새 스레드가 중복 생성될 수 있는 가능성이 존재합니다. 추후 비관적 락(`@Lock`)이나 분산 락을 통한 제어가 필요합니다.
2. **OpenAI 스트리밍 파싱 정교화**: 현재는 `delta.content`만 추출하는 형태입니다. 향후 `[DONE]` 신호 처리 및 HTTP 429/401 에러 세부 파싱 로직 추가가 필요합니다.
3. **컨텍스트 압축 전략**: 현재 토큰 낭비 방지를 위해 최근 20개 히스토리만 조회합니다.
4. **Audit 필드 활용 최소화**: 현재 세션 만료 시간 연장에 `@LastModifiedDate`(updatedAt)를 활용 중이나, 비즈니스 로직과의 책임 분할을 위해 추후 별도의 `lastActivityAt` 필드 도입이 권장됩니다.
5. **스트리밍 도중 삭제 레이스 컨디션**: 스트리밍 중인 스레드를 삭제할 시, 백그라운드 스레드에서 저장 완료 시점에 예외가 발생할 수 있습니다. 현재 로깅(`log.error`)으로 서버다운을 방어 중이나 추후 안전한 상태 관리가 필요합니다.
6. **Soft Delete 미적용 (Hard Delete 구현)**: 추후 AI 답변 품질 분석이나 유저의 복구 요청 처리를 위해 논리적 삭제(Soft Delete)가 권장되나, MVP 구현 시간 제약으로 인해 물리적 삭제(Hard Delete)로 구현되어 있습니다. 향후 `@SQLDelete` 및 `is_deleted` 필드를 도입한 리팩토링이 필요합니다.
7. **비스트리밍 및 동적 정렬 미지원**: 대화 생성 시 `isStreaming` 필드를 입력받지만 현재 MVP는 무조건 스트리밍 응답(SSE)으로 동작하도록 강제되어 있습니다. 또한, 커서 기반 페이지네이션의 특성상 대화 목록 조회 시 동적인 정렬 방향(오름차순/내림차순) 전환이 미구현되어 있습니다. 향후 방향별 커서 쿼리 추가 및 분기 처리가 필요합니다.
8. **테스트 코드 미구현**: 제한된 시간에 MVP를 구현하기 위해 테스트코드 작성이 생략되어 있습니다. 테스트 전략 수립과 테스트 코드 작성이 필요합니다.
